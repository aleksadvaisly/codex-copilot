# GitHub Copilot Native Provider Plan For Codex -- Checklist

| ID    | Requirement                                                                                                  | Acceptance Criteria                                                                                                                                                                                                                       |
| ----- | ------------------------------------------------------------------------------------------------------------ | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| REQ1  | Register `github-copilot` as a built-in provider in `codex-model-provider-info`.                             | `built_in_model_providers(/* openai_base_url */ None)` contains `github-copilot`; crate tests verify provider ID and metadata.                                                                                                            |
| REQ2  | Copilot provider metadata must be provider-correct, not OpenAI-auth-shaped.                                  | Built-in Copilot provider uses `wire_api = responses`, `requires_openai_auth = false`, and an explicit `supports_websockets` MVP default; tests assert those fields.                                                                      |
| REQ3  | Implement native Copilot device authentication and local token persistence inside the repo.                  | Rust module or crate exists for Copilot auth; unit tests cover device-code request, poll completion, persistence reload, and invalid credential handling.                                                                                 |
| REQ4  | Add a native runtime auth path for Copilot bearer retrieval that is independent from `requires_openai_auth`. | Runtime auth code can request a Copilot bearer token without OpenAI login mode; tests cover token reuse and unauthorized-token invalidation/retry behavior.                                                                               |
| REQ5  | Implement native Copilot model discovery parsing.                                                            | Typed serde structs parse a captured or fixture-based Copilot `/models` payload without defaulting to unstructured `Value`; tests cover normal, missing-field, and unknown-field cases.                                                   |
| REQ6  | Translate Copilot model catalog entries into Codex `ModelInfo` with centralized mapping policy.              | A provider-specific translator produces `ModelInfo` values with explicit mapping rules for `slug`, `display_name`, `description`, reasoning levels, visibility, shell/tool capability metadata, and defaults; unit tests verify mappings. |
| REQ7  | Integrate Copilot model discovery into `models-manager` so `/model` works natively.                          | `github-copilot` model refresh path populates translated models into the existing cache and picker flow; integration tests cover initial refresh, cached reuse, and refresh failure behavior.                                             |
| REQ8  | Add the minimum provider-specific request policy needed for Copilot streaming turns.                         | Provider-specific request customization hook exists; non-Copilot providers remain unchanged; tests cover Copilot header injection and one streamed text-turn path.                                                                        |
| REQ9  | Add provider-aware Copilot login and status UX without a full generic auth rewrite.                          | When `github-copilot` is active and credentials are missing or invalid, TUI/app-server flows present Copilot-specific login and error/status copy; tests verify provider-specific messaging.                                              |
| REQ10 | Document and validate the native Copilot provider end-to-end.                                                | Docs explain provider selection, credential storage, MVP limitations, and enterprise status; targeted tests pass for changed Copilot crates and no existing provider regressions are introduced.                                          |

## Known Traps

| TRA-ID | REQ-ID           | Trap                                                                                                                                    | Category             | Mitigation                                                                                                                                                           |
| ------ | ---------------- | --------------------------------------------------------------------------------------------------------------------------------------- | -------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| TRA1   | REQ1             | Registering a built-in provider with an unverified request-path contract can create a provider that is selectable but not yet usable.   | Hidden Prerequisite  | Limit the first slice to provider registration metadata only, keep request/auth integration separate, and document that runtime compatibility is pending later REQs. |
| TRA2   | REQ2, REQ9       | Existing onboarding and runtime flows use `requires_openai_auth` as a proxy for hosted-provider login needs.                            | Hidden Prerequisite  | Keep Copilot out of `requires_openai_auth`, then add explicit provider-aware login handling in a later slice instead of overloading OpenAI-specific logic.           |
| TRA3   | REQ3, REQ4       | Copilot auth may require multiple token forms or refresh semantics that do not match the existing external bearer flow.                 | Boundary/Mismatch    | Build a dedicated native Copilot auth module with typed storage and explicit invalidation semantics before wiring it into generic runtime request code.              |
| TRA4   | REQ5, REQ6, REQ7 | Copilot `/models` schema does not match Codex `ModelsResponse` and some required `ModelInfo` fields have no direct upstream equivalent. | Implementation Depth | Parse raw Copilot responses in a provider-specific module and centralize default/mapping policy in one translator instead of scattering field fallbacks.             |
| TRA5   | REQ8             | Copilot may require dynamic per-request headers or request shaping that static provider config cannot express.                          | Hidden Prerequisite  | Add a provider-keyed request customization hook only after a plain Responses path is proven insufficient.                                                            |
| TRA6   | REQ10            | Full TUI or workspace-wide verification is expensive and can obscure progress on low-risk slices.                                       | Hidden Prerequisite  | Use crate-scoped tests and iteration-specific review artifacts, then defer broader end-to-end validation until auth and model discovery are implemented.             |
